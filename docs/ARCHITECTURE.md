## Временная шкала бекапов

**Стратегия для домашнего сервера:**

```
daily/ (хранится 7 дней)
├── nextcloud-20260315.sql.gz   ←── Последний
├── nextcloud-20260314.sql.gz
├── nextcloud-20260313.sql.gz
├── nextcloud-20260312.sql.gz
├── nextcloud-20260311.sql.gz
├── nextcloud-20260310.sql.gz
└── nextcloud-20260309.sql.gz   ←── Самый старый (удалится завтра)

weekly/ (хранится 4 недели)
├── nextcloud-202611.sql.gz     (неделя 11)
├── nextcloud-202610.sql.gz     (неделя 10)
├── nextcloud-202609.sql.gz     (неделя 9)
└── nextcloud-202608.sql.gz     (неделя 8)

monthly/ (хранится 12 месяцев)
├── nextcloud-202603.sql.gz     (март)
├── nextcloud-202602.sql.gz     (февраль)
├── nextcloud-202601.sql.gz     (январь)
└── nextcloud-202512.sql.gz     (декабрь прошлого года)

last/ (последние 2 бекапа)
├── nextcloud-20260315-031400.sql.gz
└── nextcloud-20260315-031300.sql.gz
```

### Расписание бекапов

**Для разных типов приложений:**

```yaml
# Критичные данные (Bitwarden, Nextcloud)
SCHEDULE: "0 */2 * * *"    # Каждые 2 часа
BACKUP_KEEP_DAYS: 14       # 2 недели

# Обычные приложения (WordPress, Gitea)
SCHEDULE: "0 2 * * *"      # Раз в день в 02:00
BACKUP_KEEP_DAYS: 7        # 1 неделя

# Некритичные/тестовые
SCHEDULE: "0 2 * * 0"      # Раз в неделю (воскресенье)
BACKUP_KEEP_DAYS: 30       # 1 месяц
```

### Особенности для домашнего сервера:

- ✅ Остальные приложения **продолжают работать** во время восстановления
- ✅ PostgreSQL остаётся запущенным
- ✅ Только целевое приложение останавливается
- ✅ Минимальный downtime (~30 секунд)

### Рекомендации по организации:

**Папка `volumes/`:**
- Единое место для всех persistent данных
- Легко делать полный бекап всего сервера
- Должна быть на отдельном диске/разделе

**Папка `docker/`:**
- Конфигурации контейнеров
- Эти файлы идут в git (без секретов!)

**Папка `scripts/`:**
- Автоматизация рутинных задач
- Мониторинг, обновления, бекапы

## Best Practices для домашнего сервера

### 1. Регулярное обслуживание

```bash
# Еженедельно (воскресенье в 03:00)
0 3 * * 0 cd /path/to/server && docker-compose pull
0 3 * * 0 cd /path/to/server && docker-compose down && docker-compose up -d
```

### 2. Мониторинг места на диске

```bash
# Ежедневно в 06:00
0 6 * * * df -h /var/lib/docker | mail -s "Disk Usage" you@email.com
```

### 3. Тестирование бекапов

```bash
# Ежемесячно проверяйте восстановление
./restore-backup-multi.sh nextcloud
```

### 4. Offsite бекапы

Настройте синхронизацию на внешнее хранилище:
- rclone + Google Drive/Dropbox
- rsync на другой сервер
- Synology/QNAP Cloud Sync

## Одна БД PostgreSQL для нескольких приложений

```
┌─────────────────────────────────────────────────────────────┐
│                    PostgreSQL Instance                      │
│                        :5432                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐    │
│  │ DB: nextcloud │  │ DB: wordpress │  │ DB: customapp │    │
│  │ User: ncuser  │  │ User: wpuser  │  │ User: cuuser  │    │
│  └───────┬───────┘  └───────┬───────┘  └───────┬───────┘    │
│          │                  │                  │            │
└──────────┼──────────────────┼──────────────────┼────────────┘
           │                  │                  │
    ┌──────▼──────┐    ┌──────▼──────┐    ┌─────▼──────┐
    │ NC Backup   │    │ WP Backup   │    │ CA Backup  │
    │ Service     │    │ Service     │    │ Service    │
    └──────┬──────┘    └──────┬──────┘    └─────┬──────┘
           │                  │                 │
           ▼                  ▼                 ▼
    volumes/backup/pg/nextcloud/
    volumes/backup/pg/wordpress/
    volumes/backup/pg/customapp/
```

**Преимущества:**

- ✅ Меньше потребление ресурсов
- ✅ Проще управление
- ✅ Единая точка для мониторинга

**Недостатки:**

- ⚠️ Все приложения зависят от одного PostgreSQL
- ⚠️ Сложнее изолировать проблемы
